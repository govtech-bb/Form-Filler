# Plan: Scope host access to `.gov.bb` + localhost (CWS â€” Broad Host Permissions)

## Goal

Clear the Chrome Web Store "Broad Host Permissions" review flag by replacing the
extension's `<all_urls>` access with a concrete allowlist â€” every `.gov.bb` host
(prod + staging, any subdomain/path) plus localhost for development â€” then ship a
resubmittable build.

## Background

The *source* `manifest.json` declares a content script with
`"matches": ["<all_urls>"]`. Because the build uses `@crxjs/vite-plugin`
(`crx({ manifest })`), crxjs propagates that match pattern into **two** places in
the *built* `dist/manifest.json`, and both trigger the broad-host warning:

1. `content_scripts[0].matches` â†’ `["<all_urls>"]`
2. the auto-generated `web_accessible_resources[0].matches` â†’ `["<all_urls>"]`
   (the dynamic-import chunk for the content script)

Narrowing the matches in the source `manifest.json` fixes both at once â€” crxjs
mirrors the content-script matches onto the generated web-accessible-resource.

The content script is purely reactive: on load it only registers a
`chrome.runtime.onMessage` listener (`src/content/index.ts:118`) and does nothing
until the background messages it after a user gesture (keyboard command or popup
fill). There is no auto-fill-on-load, so scoping where it injects changes no
behaviour on the allowlisted sites.

This **supersedes** the reasoning in `remove-tabs-permission.md`, which kept
`activeTab`. With a concrete `host_permissions` allowlist, `activeTab` becomes
redundant: it existed only to grant `chrome.scripting.executeScript` temporary
host access on a gesture. `host_permissions` now authorizes that same
`executeScript` fallback (`ensureContentScript`, `src/background/index.ts:37`) on
exactly the allowlisted hosts, so `activeTab` can be dropped.

## Approach

**Narrow the declarative content script** (chosen): change the source
`manifest.json` `content_scripts[0].matches` to the allowlist, add a matching
`host_permissions`, drop `activeTab`, bump the version, and rebuild. crxjs
auto-narrows the web-accessible-resource to the same patterns. Smallest change;
keeps the existing auto-inject + `ensureContentScript` fallback flow intact.

Alternative considered â€” *fully programmatic injection* (remove `content_scripts`
entirely, inject only via `executeScript`): rejected. More moving parts with crxjs
(the loader filename would no longer live in the manifest for `ensureContentScript`
to read) with no upside now that we have a concrete allowlist.

Alternative considered â€” *appeal / justify `<all_urls>`*: rejected. A scoped
allowlist is unambiguous and faster than arguing for broad access.

## Scope

- In `manifest.json`:
  - Set `permissions` to `["scripting", "storage"]` (drop `activeTab`).
  - Add a `host_permissions` array (patterns below).
  - Change `content_scripts[0].matches` from `["<all_urls>"]` to the same
    allowlist patterns.
  - Bump `version` `1.0.1` â†’ `1.0.2` (the Web Store will not accept a re-upload at
    the same version; `manifest.json` is the build's source of truth).
- Bump `version` `1.0.1` â†’ `1.0.2` in `package.json` for consistency.
- Rebuild `dist/` with `pnpm build` so the upload package reflects the change.

### Allowlist match patterns

```json
"host_permissions": [
  "https://*.gov.bb/*",
  "http://localhost/*",
  "https://localhost/*",
  "http://127.0.0.1/*",
  "https://127.0.0.1/*"
]
```

- `https://*.gov.bb/*` matches `gov.bb` and every subdomain (`alpha.gov.bb`,
  `staging.alpha.gov.bb`, and any path beneath them) in a single pattern.
- Chrome match patterns ignore the port, so `http://localhost/*` covers
  `localhost:3000`, `localhost:8000`, etc. `127.0.0.1` and the `https://`
  variants are included per the dev requirement.
- `content_scripts[0].matches` uses this same list.
